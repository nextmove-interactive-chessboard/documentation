\subsection{Steuerung einer 2x2 Matrix}
Es wurde nicht direkt damit begonnen, die komplette Steuerung der finalen 8x8 Matrix zu programmieren, 
weil es einfacher ist, klein anzufangen und den dann vorhandenen Code auf die finale Anzahl von Feldern zu 
erweitern. Außerdem hat Christoph noch die Aufgabe, die größere 8x8 Matrix zu designen und dann 
infolgedessen zu bestellen. So bleibt zunächst nur die Möglichkeit, mit der 2x2 Matrix zu arbeiten.

\begin{scriptsize}
\begin{lstlisting}[style=Python, language=Python, captionpos=b, stepnumber=5, caption = {Erstes Beispiel für die Steuerung einer 2x2 Matrix}]
import RPi.GPIO as GPIO
import time

GPIO.setwarnings(False)
GPIO.setmode(GPIO.BCM)


pins = {
    "upPower": 17,
    "lowPower": 27,
    "left": 23,
    "right": 24,
}


# Setup
for pin in pins.values():
    GPIO.setup(pin, GPIO.OUT)

def scan_matrix():
    # --- Reihe 1 aktiv ---
    GPIO.output(pins["upPower"], GPIO.HIGH)
    GPIO.output(pins["lowPower"], GPIO.LOW)

    if GPIO.input(pins["left"]) == GPIO.LOW:
        print("A1 erkannt")
    elif GPIO.input(pins["right"]) == GPIO.LOW:
        print("A2 erkannt")

    # --- Reihe 2 aktiv ---
    GPIO.output(pins["upPower"], GPIO.LOW)
    GPIO.output(pins["lowPower"], GPIO.HIGH)

    if GPIO.input(pins["left"]) == GPIO.LOW:
        print("B1 erkannt")
    elif GPIO.input(pins["right"]) == GPIO.LOW:
        print("B2 erkannt")
try:
    while True:
        scan_matrix()
        time.sleep(0.4)

except KeyboardInterrupt:
	print("\nProgramm beendet")

finally:
	GPIO.cleanup()

\end{lstlisting}
\end{scriptsize}

Dieser Code funktioniert so, dass die beiden Reihen der Matrix abwechselnd mit Strom versorgt werden. 
Wenn eine Reihe mit Strom versorgt wird, und ein Sensor in dieser Reihe einen Magnet erkennt, dann wird 
dies von diesem Sensor als LOW Signal ausgegeben.

Ein Beispiel: Die erste Reihe wird mit Strom versorgt und der Code \glqq hört\grqq{}, ob er auf der linken 
oder rechte Reihe (Pin 23 oder 24) ein LOW Signal erhält. Wird der Sensor auf das Feld A1 (Pin 23) mit Hilfe 
eines Magneten auf LOW gesetzt, wird dies erkannt und \glqq A1 erkannt\grqq{} wird ausgegeben. 
Dasselbe passiert, wenn der Sensor auf das Feld A2 (Pin 24) mit Hilfe eines Magneten auf LOW gesetzt wird, 
nur dass hier \glqq A2 erkannt\grqq{} ausgegeben wird.

Dieses Prinzip kann dann nicht nur für die 2x2 Matrix, sondern auch für die finale 8x8 Matrix angewendet 
werden. Der Raspberry Pi schaltet dann nicht mehr nur zwischen zwei Reihen, sondern zwischen acht Reihen um. 
Je nachdem, welche Reihe mit Strom versorgt wird, wird überprüft, ob auf einem der acht Sensoren in dieser 
Reihe ein LOW Signal ausgegeben wird.

\subsubsection{Code für KY-040}
Der erste code sollte die Funktionalität sicherstellen und eine Vertrautheit mit dem Bauteil und dessen 
handling ermöglichen.
Auf der Dokumentationsseite von gpiozero findet sich unter Kapitel 2.21 \glqq Rotary encoder\grqq{} ein 
einfaches Beispiel mit einer LED. Auf Basis dieses Beispiels und dem weiteren lesen der Dokumentation von 
gpiozero, kann ein einfaches Skript erstellt werden.

\begin{scriptsize}
\begin{lstlisting}[style=Python, language=Python, captionpos=b, stepnumber=5, caption = {Erstes Beispiel für die Einbindung eines Rotary encoders}]
import threading

from gpiozero import RotaryEncoder, Button


class Encoder:
    def __init__(self):
        self.encoder = RotaryEncoder(13, 19, wrap = True,
									max_steps=100,
									bounce_time=0.05)

        self.button = Button(16, pull_up = True, 
							bounce_time = 0.05,
							hold_time = 2)

        self.done = threading.Event()

        self.button.when_released = self.handle_button  
        self.encoder.when_rotated = self.handle_rotate


    def handle_rotate(self):
        print(self.encoder.steps)


    def handle_button(self):
        print("Button released")

	
if __name__ == "__main__":
    encoder = Encoder()
    encoder.done.wait()
\end{lstlisting}
\end{scriptsize}

\subsection{Entwicklung eines Menüs}
Ein benutzerfreundliches Menü, welches zu Beginn des einschaltens erscheint, ist essenziell für ein gutes 
Spielerlebnis. Eine erste Überlegung, wie die Struktur dieses Spielmenüs aussehen soll, ist wie folgt 
aufgebaut:
Wird das Schachbrett gestartet, wird auf einem Display, welches sich auf dem Schachbrett befindet, 
angezeigt, dass sich der Benutzer zwischen  \glqq Offlinespiel\grqq{} oder \glqq Onlinespiel\grqq{} 
entscheiden kann. 


\subsubsection{\glqq Offlinespiel\grqq{}}
Wird ersteres gewählt, so erscheint ein Untermenü. Hier besteht die Wahl zwischen 
\glqq Spieler vs. Spieler\grqq{} und \glqq Spieler vs. COM\grqq{}.
\glqq Spieler vs. Spieler\glqq{} steht dafür, dass der Benutzer gegen einen Gegenüber Schach spielen kann, 
ganz ohne Einmischung eines Schachcomputers. Im Falle der Entscheidung auf diese Möglichkeit, startet das 
Spiel sofort. Die Auswahl \glqq Spieler vs. COM\grqq{} hingegen erlaubt es, noch vor dem Beginn des Spiels 
zwischen eine von drei Schwierigkeitsstufen — Leicht, Mittel oder Schwer — zu wählen, welche der 
Schachcomputer haben soll. Die Stufen unterscheiden sich davon, wie lange sich der Schachcomputer Zeit 
lässt, um einen optimalen Zug zu berechnen.
Nachdem eine Schwierigkeitsstufe gewählt wurde, geht nun das Spiel los und der Nutzer kann spielen.


\subsubsection{\glqq Onlinespiel\grqq{}}
Wird sich anstelle des Einzelspiels für das Onlinespiel entschieden, erscheint ein Untermenü, in dem der 
Benutzer aufgefordert wird, sich entwerder einzulogen oder einen neuen account zu registrieren. Ersteres 
geht davon aus, dass der Benutzer schon über ein Bentzerkonto verfügt und deshalb nur noch den NFC-Chip 
an das Lesegerät auf dem Schachbrett legen soll, damit der Benutzer in der Datenbank verglichen werden kann. 
Kann kein bereits bestehender Nutzer dem NFC-Chip zugeordnet werden, so wird der Nutzer infolgedessen 
aufgefordert, sich zu registrieren. Beim registrieren wird ein neuer Eintrag in der Datenbank erstellt, 
und der Benutzer kann fortfahren.
Im Falle des Onlinespiels kann der Spieler nur gegen einen Schachcomputer spielen und im Anschluss daran 
seine Partie auf der Website analysieren.

Ein Problem, welches noch immer auftritt, ist, dass die Auswahl zwischen den Computerschwierigkeitsstufen noch nicht 
optimal funktioniert. Bis jetzt sieht das Kriterium, wie der Code den aktuell ausgewählten Schwierigkeitsgrad erkennt, 
so aus, dass nur die Zahlen 1, 2 und 3 als gültige Werte für die Variable \glqq ai\char`_diff\grqq{} erkannt werden. 
Diese Herangehensweise ist nicht optimal und viel zu ungenau und kann deshalb in dieser Art nicht verwendet werden.

\begin{scriptsize}
\begin{lstlisting}[style=Python, language=Python, captionpos=b, stepnumber=5, caption = {Problematischer Code für die Auswahl der Computerschwierigkeitsstufe}]
case 5:  # COM difficulty selection
    if self.encoder.steps == 0:
        self.ai_diff = 1  # Easy
    elif self.encoder.steps == 1:
        self.ai_diff = 2  # Medium
    elif self.encoder.steps == 2:
        self.ai_diff = 3  # Hard
\end{lstlisting}
\end{scriptsize}

Die Lösung für dieses Problem ist es, die Auswahl der Schwierigkeitsstufe ähnlich wie die Auswahl zwischen Einzelspieler 
und Mehrspieler zu gestalten. Hier kommt der Modulus Operator von Python zum Einsatz, welcher den Rest einer Division 
zurückgibt. In diesem Fall wird die Anzahl der Schritte des Encoders aber nicht durch 2, sondern durch 3 geteilt, da 
es drei Schwierigkeitsstufen gibt. Die Ausgabe des Modulus Operators kann dann nur die Werte 0, 1 und 2 annehmen, welche 
dann jeweils einer Schwierigkeitsstufe zugeordnet werden können.

\begin{scriptsize}
\begin{lstlisting}[style=Python, language=Python, captionpos=b, stepnumber=5, caption = {Verbesserter Code für die Auswahl der Computerschwierigkeitsstufe}]
case 5:  # COM difficulty selection
    if self.encoder.steps % 3 == 0:
        self.ai_diff = 1  # Easy
    elif self.encoder.steps % 3 == 1:
        self.ai_diff = 2  # Medium
    elif self.encoder.steps % 3 == 2:
        self.ai_diff = 3  # Hard
\end{lstlisting}
\end{scriptsize}


\subsection{Spiellogik}
Die Spiellogik ist der Kern dieser Aufgabe. Das Schwierigste ist es, das Erkennen von angehobenen Figuren und das 
infolgedessen Erkennen von gültigen Zügen anzuzeigen.

\subsubsection{Das Einzelspiel (Spieler gegen Computer)}
Der Erste Teil der Spiellogik, welcher implementiert werden soll, ist die Logik für das Einzelspiel, also das Spiel 
gegen einen lokal auf dem Schachbrett installierten Schachcomputer. Hierfür wird die Python Bibliothek 
\verb|python-chess| verwendet, welche eine Vielzahl von Funktionen bietet, um nicht nur die Spiellogik eines 
Schachspiels zu implementieren, sondern auch um den Schachcomputer zu konfigurieren und Züge zu berechnen. Es wird 
sich dazu entschieden, die Stockfish Engine als Chesscomputer zu verwenden. Stockfish ist eine der stärksten und 
beliebtesten Schach-Engines der Welt und bietet die Möglichkeit, die Schwierigkeit einzustellen.

Außerdem wird für das Aufleuchten der LEDs, welche die angehobenen Figuren und die gültigen Züge anzeigen, die Python 
Bibliothek \verb|neopixel| verwendet, welche die Ansteuerung der LEDs auf der Matrix mit Hilfe des DTA Pins 
ermöglicht.

Des weiteren kommt natürlich auch die Python Bibliothek \verb|gpiozero| zum Einsatz, um die 16 Pins, welche 
jeweils für das Schalten und Lesen der Sensoren auf den Feldern zuständig sind, anfangs zu konfigurieren.

Direkt nach der Auswahl der Schwierigkeitsstufe über das Menü, wird die Stockfish Engine mit der entsprechenden 
Schwierigkeitsstufe konfiguriert. \verb|self.engine.configure| ist die Funktion, welche die Konfiguration der Engine 
ermöglicht. Hier wird die Option \glqq Skill Level\grqq{} verwendet, welche die Schwierigkeit der Engine einstellt. 
Die Werte für diese Option reichen von 0 bis 20, wobei 0 die einfachste und 20 die schwierigste Einstellung ist.

\begin{scriptsize}
\begin{lstlisting}[style=Python, language=Python, captionpos=b, stepnumber=5, caption = {Anfängliche Verarbeitung der Auswahl der Computerschwierigkeitsstufe}]
if self.comDiff == 1:  # Configuration of engine skill level
    self.engine.configure({"Skill Level": 1})  # Easy
elif self.comDiff == 2:
    self.engine.configure({"Skill Level": 5})  # Medium
elif self.comDiff == 3:
    self.engine.configure({"Skill Level": 10})  # Hard
\end{lstlisting}
\end{scriptsize}

Anstelle von \verb|Skill Level| wäre es stattdessen möglich, die Suchtiefe der Engine mit der Option 
\verb|depth={belieb. Zahl}| und der Rechenzeit \verb|time={belieb. Zahl}| einzustellen. Das macht Stockfish zwar 
schwächer, weil er weniger in die Zukunft rechnet, doch übersieht daruch aber auch oft keine offensichtlichen Fehler. 
Deshalb kann er trotz wenig Zeit und Tiefe, selbst auf leichtester Stufe zu schwer wirken. Die Einstellung über 
\verb|Skill Level| lässt Stockfish absichtlich menschliche Fehler spielen, wenn das Level niedrig ist.

Nachdem dies erledigt ist, werden die Pins für die Matrix konfiguriert. Es werden acht Pins für die Reihen als 
\verb|DigitalOutputDevices| verwendet. Diese Pins sind für das Schalten der Reihen zuständig. Die restlichen acht 
Pins sind für die Spalten als \verb|DigitalInputDevices| geschalten. Sie dienen dem lesen der Sensoren. Wird an einem 
Sensor ein Magnet erkannt, so wird von diesem Pin ein LOW Signal ausgegeben.

\begin{scriptsize}
\begin{lstlisting}[style=Python, language=Python, captionpos=b, stepnumber=5, caption = {Setzen von Schalte- und Lesepins für die Matrix}]
# Matrix-Setup
self.rowPins = [4, 17, 27, 22, 0, 5, 6, 26]  # Turned on/off by script
self.colPins = [14, 15, 23, 24, 20, 21, 7, 1]  # Process signals
self.tiles = ["H", "G", "F", "E", "D", "C", "B", "A"]

self.rows = []
for p in self.rowPins:
    self.rows.append(
        DigitalOutputDevice(p, active_high=True, initial_value=False)
    )

self.cols = []
for p in self.colPins:
    self.cols.append(DigitalInputDevice(p, pull_up=True))
\end{lstlisting}
\end{scriptsize}

Die erste Funktion in der Klasse \verb|COMGame| ist die Funktion \verb|calibrateBoard(self)|. Diese hat den Zweck, den 
aktuellen Zustand des Schachbretts einmal komplett einzulesen und als Grundlage für spätere Überprüfungen zu verwenden. 
Sie schaltet eine Reihe nach der anderen ein und überprüft dabei eine Spalte nach der anderen, ob ein LOW Signal 
ausgegeben wird. Wenn dies der Fall ist, dann werden die entsprechenden Felder in das Dictionary \verb|previousSensor| 
geschrieben, welches vorher in der \verb|__init()__| Funktion erstellt wurde. Dieses Dictionary wird später dazu verwendet, um zu überprüfen, ob eine Figur angehoben wurde oder nicht. 

\begin{scriptsize}
\begin{lstlisting}[style=Python, language=Python, captionpos=b, stepnumber=5, caption = {Funktion zum Kalibrieren des Schachbretts}]
def calibrateBoard(self):
    for r, row in enumerate(self.rows):
        row.on()
        time.sleep(self.rowChargeTime)
        rank = r + 1
        for c, col in enumerate(self.cols):
            field = self.tiles[c] + str(rank)
            if col.value == 0:
                self.previousSensor[field] = 0
            else:
                self.previousSensor[field] = 1
        row.off()
        time.sleep(self.rowDischargeTime)
\end{lstlisting}
\end{scriptsize}

\end{lstlisting}
\end{scriptsize}

Aus diesem Auschnitt ist ersichtlich, dass die Felder H1 bis A2 mit einer 1 markiert sind. Das bedeutet, dass auf diesen 
Feldern eine Figur steht.


\subsubsection{Der Mehrspieler (Spieler gegen Spieler)}
Der zweite Teil der Spiellogik ist die Logik für das Mehrspiel, also das Spiel gegen einen Gegenüber. Dieser Teil ist 
deutlich einfacher zu implementieren, da hier keine Computer-Logik implementiert werden muss. Es kann der bestehende Code für 
das Einzelspiel als Grundlage verwendet werden, da die Vorangehensweise für das Erkennen von angehobenen Figuren und 
gültigen Zügen die gleiche ist. Die Logik für den Schachcomputer muss entfernt werden.

Der Mehrspielmodus ermöglicht es, sein Spiel aufzuzeichnen. Das wird mit Hife von \verb|.pgn|-Dateien gemacht. 
\verb|pgn| steht hierbei für \glqq Portable Game Notation\grqq{} und ist ein Standardformat für die Aufzeichnung von 
Schachpartien. Die Erstveröffentlichung war im Jahr 1993 durch Steven J. Edwards, welcher das Format entwickelt hat. 
Die Züge an sich werden in verkürzter algebraischer Notation aufgezeichnet und Figurenbezeichnungen auf englisch 
verwendet. Die standartisierte Aufzeichnung von Partien im \verb|.pgn|-Format ermöglicht es, die Partien später auf der 
Website zu analysieren.

Dieses Format wird auch von der \verb|python-chess| Bibliothek unterstützt.
\begin{scriptsize}
\begin{lstlisting}[style=Python, language=Python, captionpos=b, stepnumber=5, caption = {Erstellen der PGN-Header}]
# PGN Setup
self.year = time.strftime("%Y")
self.month = time.strftime("%m")
self.day = time.strftime("%d")
self.pgn_game = chess.pgn.Game()
self.pgn_game.headers["Event"] = "Lokales Multiplayer Spiel"
self.pgn_game.headers["Date"] = f"{self.year}.{self.month}.{self.day}"
self.pgn_game.headers["White"] = "Spieler 1"
self.pgn_game.headers["Black"] = "Spieler 2"
self.pgn_node = self.pgn_game
\end{lstlisting}
\end{scriptsize}

Eine \verb|.pgn|-Datei enthält ganz oben einen Header, welcher Informationen über die Partie enthält. In der in diesem 
Projekt ausgeführten Version werden die Informationen über das Event, das Datum und die Damen der Spieler übergeben.
