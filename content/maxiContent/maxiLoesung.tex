\subsection{Steuerung einer 2x2 Matrix}
Es wurde nicht direkt damit begonnen, die komplette Steuerung der finalen 8x8 Matrix zu programmieren, 
weil es einfacher ist, klein anzufangen und den dann vorhandenen Code auf die finale Anzahl von Feldern zu 
erweitern. Außerdem hat Christoph noch die Aufgabe, die größere 8x8 Matrix zu designen und dann 
infolgedessen zu bestellen. So bleibt zunächst nur die Möglichkeit, mit der 2x2 Matrix zu arbeiten.\\

\begin{scriptsize}
\begin{lstlisting}[style=Python, language=Python, captionpos=b, stepnumber=5, caption={Erstes Beispiel für die Steuerung einer 2x2 Matrix}, label={lst:2x2matrix}]
pins = {
    "upPower": 17,
    "lowPower": 27,
    "left": 23,
    "right": 24,
}

# Setup
for pin in pins.values():
    GPIO.setup(pin, GPIO.OUT)

def scan_matrix():
    # --- Reihe 1 aktiv ---
    GPIO.output(pins["upPower"], GPIO.HIGH)
    GPIO.output(pins["lowPower"], GPIO.LOW)

    if GPIO.input(pins["left"]) == GPIO.LOW:
        print("A1 erkannt")
    elif GPIO.input(pins["right"]) == GPIO.LOW:
        print("A2 erkannt")

    # --- Reihe 2 aktiv ---
    GPIO.output(pins["upPower"], GPIO.LOW)
    GPIO.output(pins["lowPower"], GPIO.HIGH)

    if GPIO.input(pins["left"]) == GPIO.LOW:
        print("B1 erkannt")
    elif GPIO.input(pins["right"]) == GPIO.LOW:
        print("B2 erkannt")

while True:
    scan_matrix()
    time.sleep(0.4)
\end{lstlisting}
\end{scriptsize}


Dieser Code funktioniert so, dass die beiden Reihen der Matrix abwechselnd mit Strom versorgt werden. 
Wenn eine Reihe mit Strom versorgt wird, und ein Sensor in dieser Reihe einen Magnet erkennt, dann wird 
dies von diesem Sensor als LOW Signal ausgegeben.

Ein Beispiel: Die erste Reihe wird mit Strom versorgt und der Code \glqq hört\grqq{}, ob er auf der linken 
oder rechte Reihe (Pin 23 oder 24) ein LOW Signal erhält. Wird der Sensor auf das Feld A1 (Pin 23), mithilfe 
eines Magneten, auf LOW gesetzt, wird dies erkannt und \glqq A1 erkannt\grqq{} wird ausgegeben. 
Dasselbe passiert, wenn der Sensor auf dam Feld A2 (Pin 24) aufgrund eines Magneten auf LOW gesetzt wird, 
nur dass hier \glqq A2 erkannt\grqq{} ausgegeben wird.

Hierfür wurde noch die RPi.GPIO \cite{RPi.GPIO} Bibliothek verwendet. Doch da diese Bibliothek seit dem 02.06.2022 
nicht mehr aktualisiert wurde und auch kein SPI beziehungsweise I2C unterstützt, wird stattdessen ab sofort die 
gpiozero \cite{gpiozero} Bibliothek verwendet. 

Dieses Prinzip kann dann nicht nur für die 2 × 2 Matrix, sondern auch für die finale 8 × 8 Matrix angewendet werden. 
Der Raspberry Pi schaltet dann nicht mehr nur zwischen zwei Reihen, sondern zwischen acht Reihen um. 
Je nachdem, welche Reihe mit Strom versorgt wird, wird überprüft, ob auf einem der acht Sensoren in dieser Reihe ein 
LOW Signal ausgegeben wird.

\subsubsection{Code für KY-040}
Der erste code sollte die Funktionalität sicherstellen und eine Vertrautheit mit dem Bauteil und dessen Handling 
ermöglichen. Auf der Dokumentationsseite von gpiozero findet sich unter Kapitel 2.21 \glqq Rotary encoder\grqq{} 
\cite{gpiozero} ein einfaches Beispiel mit einer LED. Auf Basis dieses Beispiels und dem weiteren Lesen der 
Dokumentation von gpiozero, kann ein einfaches Skript erstellt werden.\\

\begin{scriptsize}
\begin{lstlisting}[style=Python, language=Python, captionpos=b, stepnumber=5, caption = {Erstes Beispiel für die Einbindung eines Rotary encoders}]
import threading

from gpiozero import RotaryEncoder, Button


class Encoder:
    def __init__(self):
        self.encoder = RotaryEncoder(13, 19, wrap = True,
									max_steps=100,
									bounce_time=0.05)

        self.button = Button(16, pull_up = True, 
							bounce_time = 0.05,
							hold_time = 2)

        self.done = threading.Event()

        self.button.when_released = self.handle_button  
        self.encoder.when_rotated = self.handle_rotate


    def handle_rotate(self):
        print(self.encoder.steps)


    def handle_button(self):
        print("Button released")

	
if __name__ == "__main__":
    encoder = Encoder()
    encoder.done.wait()
\end{lstlisting}
\end{scriptsize}

Dieses Skript initialisiert unter \verb|self.encoder| den drehbaren Teil des Encoders. Pin 13 und Pin 19 entsprechen 
dabei den verwendeten Pins für die jeweiligen Signale A und B. Bei dem Bauteil KY-040 werden diese Kontake nicht als 
\glqq A\grqq{} und \glqq B\grqq{} bezeichnet, sondern als \glqq CLK\grqq{} und \glqq DT\grqq{}. Die Funktion bleibt 
allerdings die gleiche. Wird der Encoder gedreht, so wird die Funktion \verb|.when_rotated| aufgerufen. Diese gibt 
integerzahlen zurück. Diese werden dann in der Funktion \verb|handle_rotate| über dem print-Statement in der Konsole 
des Raspberry Pi's ausgegeben.\\

Der Button des Encoders befindet sich auf Pin 16 und ist softwareseitig mit einem Pull-Up Widerstand konfiguriert. 
Die Option \glqq bounce\char`_time\grqq{} sorgt dafür, dass die Signale des Encoders entprellt werden, damit es nicht 
zu ungewollten Mehrfachauslösungen kommt. \glqq hold\char`_time\grqq{} ist jene Zeit in Sekunden, welche der Button 
gehalten werden muss, damit eine \verb|.when_released| Funktion ausgelöst wird. Ist dies der Fall, so wird über die 
aufgerufene Funktion \verb|handle_button| die Ausgabe \glqq Button released\grqq{} in der Konsole ausgegeben.

\subsection{Entwicklung eines Menüs}
Ein benutzerfreundliches Menü, welches zu Beginn des einschaltens erscheint, ist essenziell für ein gutes 
Spielerlebnis. Eine erste Überlegung, wie die Struktur dieses Spielmenüs aussehen soll, ist wie folgt 
aufgebaut:
Wird das Schachbrett gestartet, wird auf einem Display, welches sich auf dem Schachbrett befindet, 
angezeigt, dass sich der Benutzer zwischen \glqq Offlinespiel\grqq{} oder \glqq Onlinespiel\grqq{} 
entscheiden kann.


\subsubsection{Das Einzelspiel}
Wird ersteres gewählt, so erscheint ein Untermenü. Hier kann man die Schwierigkeitsstufe des Schachcomputers einstellen. 
Die Wahl besteht zwischen drei Stufen: Leicht, Mittel und Schwer. Die Stufen unterscheiden sich davon, wie lange sich 
der Schachcomputer Zeit lässt, um einen optimalen Zug zu berechnen. Je länger er sich Zeit lässt, desto stärker ist er, 
da er mehr Züge in die Zukunft berechnen kann. Nachdem eine Schwierigkeitsstufe gewählt wurde, geht nun das Spiel los 
und der Nutzer kann spielen. Die Partie wird nicht aufgezeichnet und nach Beendigung des Spiels kehrt die Software 
wieder automatisch zum Hauptmenü zurück, damit der Nutzer die Möglichkeit hat, eine neue Partie zu starten.


\subsubsection{Der Mehrspieler}
Wird sich anstelle des Einzelspiels für den Mehrspieler entschieden, erscheint ein Untermenü, in dem der 
Benutzer aufgefordert wird, sich entweder einzulogen oder einen neuen Account zu registrieren. Ersteres 
geht davon aus, dass der Benutzer schon über ein Bentzerkonto verfügt und deshalb nur noch den NFC-Chip 
an das Lesegerät auf dem Schachbrett legen soll, damit der Spieler in der Datenbank verglichen werden kann. 
Kann kein bereits bestehender Nutzer dem NFC-Chip zugeordnet werden, so wird der Nutzer infolgedessen 
aufgefordert, sich zu registrieren. Beim Registrieren wird ein neuer Eintrag in der Datenbank erstellt, 
und der Benutzer kann fortfahren.\\

Es ist zu beachten, dass ein Benutzer nur in der Lage ist, seine Partie im Anschluss eines beendeten Spiels auf der 
Internetseite zu analysieren, wenn er sich für den Mehrspielmodus entscheidet.

Ein Problem, welches noch immer auftritt, ist, dass die Auswahl zwischen den Computerschwierigkeitsstufen noch nicht 
optimal funktioniert. Bis jetzt sieht das Kriterium, wie der Code den aktuell ausgewählten Schwierigkeitsgrad erkennt, 
so aus, dass nur die Zahlen 1, 2 und 3 als gültige Werte für die Variable \verb|comDiff| erkannt werden. 
Diese Herangehensweise ist nicht optimal, viel zu ungenau und kann deshalb in dieser Art nicht verwendet werden.

\begin{scriptsize}
\begin{lstlisting}[style=Python, language=Python, captionpos=b, stepnumber=5, caption = {Problematischer Code für die Auswahl der Computerschwierigkeitsstufe}]
case 5:  # COM difficulty selection
    if self.encoder.steps == 0:
        self.comDiff = 1  # Easy
    elif self.encoder.steps == 1:
        self.comDiff = 2  # Medium
    elif self.encoder.steps == 2:
        self.comDiff = 3  # Hard
\end{lstlisting}
\end{scriptsize}

Die Lösung für dieses Problem ist es, die Auswahl der Schwierigkeitsstufe ähnlich wie die Auswahl zwischen Einzelspieler 
und Mehrspieler zu gestalten. Hier kommt der Modulo Operator von Python zum Einsatz, welcher den Rest einer Division 
zurückgibt. In diesem Fall wird die Anzahl der Schritte des Encoders aber nicht durch 2, sondern durch 3 geteilt, da 
es drei Schwierigkeitsstufen gibt. Die Ausgabe des Modulo Operators kann dann nur die Werte 0, 1 und 2 annehmen, welche 
dann jeweils einer Schwierigkeitsstufe zugeordnet werden können.

\begin{scriptsize}
\begin{lstlisting}[style=Python, language=Python, captionpos=b, stepnumber=5, caption = {Verbesserter Code für die Auswahl der Computerschwierigkeitsstufe}]
case 5:  # COM difficulty selection
    if self.encoder.steps % 3 == 0:
        self.comDiff = 1  # Easy
    elif self.encoder.steps % 3 == 1:
        self.comDiff = 2  # Medium
    elif self.encoder.steps % 3 == 2:
        self.comDiff = 3  # Hard
\end{lstlisting}
\end{scriptsize}

Außerdem wird für das Aufleuchten der LEDs, welche die angehobenen Figuren und die gültigen Züge anzeigen, die Python

jeweils für das Schalten und Lesen der Sensoren auf den Feldern zuständig sind, anfangs zu konfigurieren.

Direkt nach der Auswahl der Schwierigkeitsstufe über das Menü wird die Stockfish Engine mit der entsprechenden

Die Werte für diese Option reichen von 0 bis 20, wobei 0 die einfachste und 20 die schwierigste Einstellung ist.

Anstelle von \verb|Skill Level| wäre es stattdessen möglich, die Suchtiefe der Engine mit der Option

doch übersieht dadurch aber auch oft keine offensichtlichen Fehler. Deshalb kann er trotz wenig Zeit und Tiefe, 
selbst auf leichtester Stufe zu schwer wirken. Die Einstellung über \verb|Skill Level| lässt Stockfish absichtlich 
menschliche Fehler spielen, wenn das Level niedrig ist.

Sensor ein Magnet erkannt, so wird von diesem Pin ein LOW Signal ausgegeben.

Die erste Funktion in der Klasse \verb|COMGame| ist die Funktion \verb|calibrateBoard(self)|. Diese hat den Zweck, den 
aktuellen Zustand des Schachbretts einmal komplett einzulesen und als Grundlage für spätere Überprüfungen zu verwenden. 
Sie schaltet eine Reihe nach der anderen ein und überprüft dabei eine Spalte nach der anderen, ob ein LOW Signal 
ausgegeben wird. Wenn dies der Fall ist, dann werden die entsprechenden Felder in das Dictionary \verb|previousSensor| 
geschrieben, welches vorher in der \verb|__init()__| Funktion erstellt wurde. Dieses Dictionary wird später dazu 
verwendet, um zu überprüfen, ob eine Figur angehoben wurde oder nicht. 

\begin{scriptsize}
\begin{lstlisting}[style=Python, language=Python, captionpos=b, stepnumber=5, caption={Funktion zum Kalibrieren des Schachbretts}]
def calibrateBoard(self):
    for r, row in enumerate(self.rows):
        row.on()
        time.sleep(self.rowChargeTime)
        rank = r + 1
        for c, col in enumerate(self.cols):
            field = self.tiles[c] + str(rank)
            if col.value == 0:
                self.previousSensor[field] = 0
            else:
                self.previousSensor[field] = 1
        row.off()
        time.sleep(self.rowDischargeTime)
\end{lstlisting}
\end{scriptsize}

\subsubsection{Der Mehrspieler (Spieler gegen Spieler)}
Der zweite Teil der Spiellogik ist die Logik für das Mehrspiel, also das Spiel gegen eine weitere Person. Dieser Teil ist 
deutlich einfacher zu implementieren, da hier keine Computer-Logik implementiert werden muss. Es kann der bestehende 
Code für das Einzelspiel als Grundlage verwendet werden, da die Herangehensweise für das Erkennen von angehobenen 
Figuren und gültigen Zügen die gleiche ist. Die Logik für den Schachcomputer muss entfernt werden.

Der Mehrspielmodus ermöglicht es, sein Spiel aufzuzeichnen. Das wird mithilfe von \verb|.pgn|-Dateien gemacht. 
\verb|pgn| steht hierbei für\glqq Portable Game Notation\grqq{} und ist ein Standardformat für die Aufzeichnung von 
Schachpartien. Die Erstveröffentlichung war im Jahr 1993 durch Steven J. Edwards, welcher das Format entwickelt hat. 
Die Züge an sich werden in verkürzter algebraischer Notation aufgezeichnet und Figurenbezeichnungen auf Englisch 
verwendet. Mit der standartisierten Aufzeichnung von Schachpartien im \verb|.pgn|-Format wird es ermöglicht, die 
Partien später auf der Website zu analysieren.

Dieses Format wird auch von der \verb|python-chess| Bibliothek unterstützt.

\begin{scriptsize}
\begin{lstlisting}[style=Python, language=Python, captionpos=b, stepnumber=5, caption={Erstellen der PGN-Header}, label={lst:pgnHeader}]
# PGN Setup
self.year = time.strftime("%Y")
self.month = time.strftime("%m")
self.day = time.strftime("%d")
self.pgnGame = chess.pgn.Game()
self.pgnGame.headers["Event"] = "Lokales Multiplayer Spiel"
self.pgnGame.headers["Date"] = f"{self.year}.{self.month}.{self.day}"
self.pgnGame.headers["White"] = self.userCode
self.pgnGame.headers["Black"] = "Spieler 2"
self.pgnNode = self.pgnGame
self.filename = f"{self.userCode}.pgn"
\end{lstlisting}
\end{scriptsize}

Eine \verb|.pgn|-Datei enthält ganz oben einen Header, welcher Informationen über die Partie enthält. In der in diesem 
Projekt ausgeführten Version werden die Informationen über das Event, das Datum und die Namen der Spieler übergeben. 
Der Spielername von Weiß ist dabei der Code, welchen der Spieler bei der Registrierung eingegeben hat.

\subsection{Die Datenübertragung}
Nachdem die Spiellogik implementiert ist und die Mehrspielerpartien im \verb|.pgn|-Format aufgezeichnet werden, muss 
nun die Möglichkeit geschaffen werden, diese Partien nach Spielende dem Server zu übermitteln, damit sie auf der 
Website analysiert werden können. Die Datenübertragung mit Sockets ist dabei die schnellste Möglichkeit, um die Daten 
vom Schachbrett zu übertragen.
Die \verb|socket| Bibliothek von Python hat dabei alle benötigten Funktionen, um dies zu ermöglichen. Die im Anhang 
befindliche Funktion
\verb|postGame(self, boardTurn)| in multigame.py [\ref{lst:multigame} | Z. 162-168] stellt nach Erkennung eines 
Schachmatt-Zustands eine Verbindung zum Server her und übermittelt die \verb|.pgn|-Datei. Folgender Ausschnitt aus 
besagter Funktion zeigt die einzelnen Schitte, welche getätigt werden.

\newpage

\begin{scriptsize}
\begin{lstlisting}[style=Python, language=Python, captionpos=b, stepnumber=5, caption={Funktion zum Übermitteln der PGN-Datei an den Server}, label={lst:postGameSocket}]
def postGame(self, boardTurn):
...
    self.clientSocket.connect(self.serverAddress)

    with open(self.filename, "rb") as file:
        data = file.read()
        self.clientSocket.sendall(data)

    self.clientSocket.close()
\end{lstlisting}
\end{scriptsize}

Zu Beginn wird eine Verbindung zum Server hergestellt. \verb|self.serverAddress| ist ein Tupel bestehend aus der 
IP-Adresse und dem Port des Servers. Sobald die Verbindung hergestellt ist, wird mit\\\verb|with open(self.filename, "rb") as file| 
die Datei im binären Lesemodus geöffnet. Das ist deshalb wichtig, weil der Socket Bytes sendet. Textmodus \verb|r| 
alleine würde einen \verb|str| zurückgeben. Mit \verb|data = file.read()| wird der Inhalt der Datei in die Variable 
\verb|data| geschrieben.\\\verb|self.clientSocket.sendall(data)| sendet nun die Daten über die Socket-Verbindung an 
den Server. Am Ende wird die Verbindung mit\\\verb|self.clientSocket.close()| wieder geschlossen. Im Anschluss wird 
die \verb|.pgn|-Datei gelöscht.

Sollte es vorkommen, dass die Verbinung zum Server nicht hergestellt werden kann — zum Beispiel, wenn das Schachbrett 
keine Internetverbindung zum Server herstellen kann — so wird nach Beendigung der Partie das aufgenommene Spiel vom 
internen Speicher des Rapsberry Pi gelöscht. So wird verhindert, dass sich zu viele \verb|.pgn|-Dateien ansammeln.
Das geschieht in der Funktion \verb|run(self)| [\ref{lst:multigame} | Z. 287] als letztes Statement im \verb|try-except-finally|-Block.
